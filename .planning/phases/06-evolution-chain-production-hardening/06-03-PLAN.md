# Plan 06-03: Realistic-Latency Fake-Provider Concurrency Safety

## Objective

Verify harness concurrency safety only. Use 20 realistic-latency fake-provider
runs to prove artifact paths, tool-loop state, provider messages, runtime
records, and trajectory records do not cross-contaminate across simultaneous
requests. This plan does not claim real DeepSeek concurrency capacity.

## must_haves

- truths:
  - 20 fake-provider request runs can execute concurrently with fresh runtime
    and provider instances.
  - Every request writes exactly three artifacts and all artifact paths are unique.
  - Each provider message snapshot contains only its own request/session data.
  - Runtime traces and records stay scoped to the request that produced them.
  - If trajectory buffering from 06-02 is present, each trajectory references
    only its own request id and artifact paths.
- forbid:
  - Adding provider concurrency tuning, limiter, circuit breaker, retry manager, or global scheduling machinery.
  - Sharing one `ScriptedProvider` instance across concurrent requests.
  - Mutating production code solely to make a test easier.
  - Treating this fake-provider result as real API rate-limit evidence.

<threat_model>
T-06-07: concurrent runs leak messages, paths, or records across requests.
Mitigation: concurrency smoke asserts per-request provider snapshots, output dirs, artifact paths, and trajectory request ids are isolated.

T-06-08: a test accidentally hides a global-state bug by running providers without realistic latency.
Mitigation: delayed scripted provider sleeps per generate call before returning scripted turns.
</threat_model>
