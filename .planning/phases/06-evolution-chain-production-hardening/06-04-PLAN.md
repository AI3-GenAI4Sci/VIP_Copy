# Plan 06-04: Minimal Terminal Progress And DeepSeek Rate-Limit Facts

## Objective

Add CI-safe terminal progress for long fake or real runs and record current
DeepSeek rate-limit facts without tuning concurrency. The progress surface is
plain stdout by default, disabled by `--no-progress` or CI mode, and shows only
completed/total, current request or scenario, failure count, and delta trial
count.

## must_haves

- truths:
  - Progress rendering has no new third-party dependency.
  - `--no-progress` and CI mode produce plain or no progress output suitable for logs.
  - Progress output includes completed/total, current request/scenario, failure count, and delta trial count.
  - DeepSeek facts document current model, base URL, SDK retry behavior, 429 classification, optional header observations, and fact date.
  - No limiter or concurrency tuning is added in this plan.
- forbid:
  - Adding `rich`, `tqdm`, logging configuration, or a progress manager service.
  - Making network calls in normal tests.
  - Changing DeepSeek runtime defaults except to expose read-only fact extraction.
  - Claiming rate-limit headers exist when the optional probe cannot see them.

<threat_model>
T-06-09: progress output pollutes CI logs or hides failures.
Mitigation: tests cover no-progress and CI-safe rendering, and progress writes to a supplied stream.

T-06-10: rate-limit fact recording mutates provider behavior.
Mitigation: provider fact extraction is read-only and tests assert no limiter/concurrency knobs are introduced.
</threat_model>
