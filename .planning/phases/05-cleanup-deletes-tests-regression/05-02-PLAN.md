# Plan 05-02: 删除残留 hard_check gate 分支（CLEAN-03 实际命中点）

## Objective

`workflow/payloads.py:113-156` 的 `rubric_payload_for` 仍然消费
`hard_check_artifact` 并按 `passed=False / blocking_rule_ids` 过滤
candidate；`workflow/dag_runner.py` 通过 `provider_payload_for_node`
间接把 `dependency_payloads.get("hard_check")` 注入这条分支。c17 没有任何
节点产出 `hard_check` artifact，分支永远走空字典路径，是纯结构性残留
（CONTEXT.md D-06 假设的反例，RESEARCH §1 给出修订）。

本计划删除这条 gate：删 `rubric_payload_for` 的 `hard_check_artifact`
形参与过滤分支、删 `provider_payload_for_node` 里 `hard_check` 注入、
增一个负面测试断言这条分支已删除。

## must_haves

- truths:
  - `rg 'hard_check' seers_harness/' --type py` 命中 0 行
  - `rubric_payload_for` 签名只接 `scenario` 和 `copy_artifact` 两个 keyword
  - 122-test baseline 加新增的负面断言后保持绿
- forbid:
  - 改动 `tests/` 里现存的负面断言（如有 `STOP_GATE` / `polling` /
    `self_check` 否定断言，全部保留）
  - 改动 `tools/skill_tools.py` 里 `record_factor` / `record_candidate` /
    `judge_candidate` 的 runtime 校验逻辑（这些是 hand role，是 c17 真正
    的检查点，不是 hard-check gate）
